Pisces: Private and Compliable Cryptocurrency Exchange

Intro

就像股票和其他商品一样，人们在交易平台上买卖加密货币，主要是在Coinbase等中心化平台上，这些平台本质上是加密货币的市场。在那里，客户可以支付像美元这样的法定货币来获得一些硬币，例如比特币，或者将他们在平台上的硬币转移到外部账户（取款）1。尽管有去中心化交易的承诺，但出于可用性甚至监管原因，这些中心化交易平台仍然发挥着重要作用。

与传统的证券交易所一样，这些交易平台必须遵守包括了解您的客户（KYC）在内的法规。他们要求企业核实客户的身份。从本质上讲，当客户端/用户在交易平台注册帐户时，通常需要提供真实世界的身份证明文件，例如护照或印有地址的信封，以便平台进行验证。

严重的隐私威胁：尽管提供了便利，但这些中心化的加密货币交易平台引发了更严重的潜在隐私泄露问题

在交换环境中，一个更令人担忧的问题是，交易所可以被视为现实世界和加密货币世界之间的桥梁，这放大了潜在隐私泄露的影响（在交换中，包括身份和账户在内的用户记录）。用户可以从区块链上的个人账户向平台存入代币，也可以从个人账户中提取代币（转出平台）。

由于大多数区块链是透明的（除了极少数链，如Monero [32]， Zcash[8]）和可公开访问的（例如，比特币，以太坊），该平台基本上可以提取所有交易历史，知道用户的真实身份。在前一种情况下，平台立即将真实身份与他的接收地址联系起来，并追溯之前所有的链上交易；而在后一种情况下更糟糕的是，平台知道用户的真实身份，并且确切地知道用户请求提取（转移到）的加密货币的哪个账户/地址，以及所有未来的交易。

  • UTXO模型中交易透明：可以获得某个地址所有的历史交易
  • 账户模型：知道用户的真实身份，并且确切地知道用户请求提取（转移到）的加密货币的哪个账户/地址，以及所有未来的交易。

该平台可以很容易地推断出用户Alice用比特币购买了一辆特斯拉汽车，因为她从平台上取出了代币，然后将它们转移到特斯拉的比特币账户（这可能是公开的）。

现有的中心化加密货币交易所立即“破坏”了区块链的假名保护，平台可以获得大量本不应该被了解的信息，例如平台外客户的购买/交易历史。这甚至比传统的股票/商品交易所更糟糕，后者的隐私可能会在系统内被侵犯，但系统外的用户信息不会被披露。

我们希望设计一个加密货币交易系统，至少可以恢复用户的匿名性/隐私性，使外部交易记录与用户真实身份之间不可链接

外部匿名机制的不足。第一种可能的方法是保持现有的交易所不变，并通过在每个区块链上匿名进行链上支付/转账（用于硬币存款和取款）来切断交易所与外部区块链之间的联系，这样就没有人可以将付款人和收款人联系起来。不幸的是，所有这些外部匿名解决方案都是不够的。

首先，像Zcash这样的完全匿名的链上支付只支持他们自己的原生代币，而在大多数交易平台中，比特币、以太坊和许多其他加密代币是主要的交易对象，不支持。

更奇特的解决方案，如匿名和私人第二层支付解决方案[26]，[25],[35],[31]，[24]和基于智能合约的私人支付解决方案[15]，[21]也存在。

有人可能会想，我们是否可以让平台成为这些解决方案中的支付者。然而，现有的解决方案主要考虑k-匿名（其中k是一个epoch的活跃用户数量）针对[28]，[25]，[35]中的hub或[31]中的领导者或[24]中的局外人，而不是针对付款人本人。在我们的例子中，平台是付款人，并且在取币过程中确切地知道收款人的地址。

[36]和[25]最近的工作甚至考虑了一个匿名（k-匿名）的支付中心，假设一个固定的面额。除了k通常很小之外，在交易平台上的提现交易也很难是固定的。当两个用户提取不同数量的硬币时，平台可以简单地区分他们

交易平台内未开发的匿名性。以上分析提示，仅依靠外部匿名机制是不够的，需要进一步加强平台内部的匿名保护。匿名问题是在不同环境下广泛研究的经典话题，包括在加密货币中；然而，我们将证明，这些工作中的很大一部分并不适用于我们的交换制度

首先，匿名支付中心解决方案不仅不能直接适用，而且即使是技术（例如，将交易平台视为中心，而每个用户既可以是付款人也可以是收款人）也不足以实现“内部”匿名。同样，关键的区别在于支付中心（和其他平台）之间的功能差异

Contribution

  • 建模。首次正式定义了私有和可兼容的加密货币交换。我们先给出一个基本版本的匿名作为预热，它只关心提现操作。正如我们在上面简要讨论的那样，只隐藏部分事务数据可能无法提供合理的强匿名性。最后，我们定义了安全模型，坚持认为交换基本上不会向平台泄露任何信息。通过这种方式，我们获得了最好的匿名性（考虑到公开退出总是存在的）。我们还仔细定义了诸如防透支和合规性等稳健性属性。
  • 我们首先给出了一个非常简单的结构来满足基本的可撤回匿名性，并说明了它的局限性。然后，我们设计了第一个私有的、可兼容的交换系统，该系统在全私有模型下是安全的。
  • 用户隐藏在一个庞大的匿名集合中，他们不能提取比自己拥有的更多的资产，也不能报告虚假的合规信息。为了获得完全的匿名性，在每笔交易中都尽可能地隐藏用户的信息，包括用户身份和交换资产的详细信息。通过为我们的目的专门设计的有效的NIZK证明，确保了良好的性能。请注意，证明交换请求的正确性通常会导致证明的大小与资产类型的总数成线性关系；相反，我们提出了一种与系统中资产类型和用户数量无关的高效结构，其通信和计算成本都是恒定的。
  • 业绩评估。我们实现并评估系统，以测试每个操作的成本分解，并将其与普通交换（不匿名）进行比较。考虑到TLS通信的存在，我们的开销是最小的。我们还与其他相关系统进行了比较，以获得进一步的证据。

Pre

盲签名（Blind Signatures） 是一种密码学签名方案，它允许用户在不透露消息内容的情况下，向签名者请求签名。换句话说，用户可以将消息“盲化”（blind），然后向签名者请求签名，而签名者并不知道消息的内容。这种方案通常用于增强隐私保护，例如在数字支付或匿名认证中。

基本匿名提现方案

基本提现匿名性方案 ΠBWA：

这个方案涉及以下几个组件：

  • 加法同态承诺方案（Com）：这是一个支持加法同态的承诺方案，即它允许对承诺的值进行加法操作而不需要揭示原始消息的内容。
  • 盲签名方案 Πbs：包括生成密钥对、盲签名（BlindSign）、接收盲签名（BlindRcv）和验证签名（Vrfy）的过程。盲签名用于隐藏消息内容，使得签名者无法看到消息内容。
  • 零知识认证协议（ZKAoK）：这是一个协议，允许一方证明某个声明是正确的，而不泄露除声明本身外的任何信息。ZKAoK 在这个上下文中可能用于证明某些信息在匿名提现过程中是正确的。

系统初始化

（Setup(1λ) → epp）：

  • 输入： 安全参数 λ

  • 输出： 系统参数 epp，它包含所有涉及的密码学原语的公共参数，以及与系统相关的其他参数。

    具体来说，系统参数 epp 包含：

    • 公共参数： 包括加法同态承诺方案 Com、盲签名方案 Πbs等所需要的密码学参数。
    • 特定的公共系统参数：
      • 资产种类数 nn：系统中涉及的资产种类数。
      • 最大余额 vmax=p−1v_{\text{max}} = p-1vmax​=p−1：表示系统允许的最大余额，其中 p 是一个超级多项式量。
      • 动态参数： 例如每种资产 i∈[n]的当前价格 pri

    总结：epp是一个包括所有相关密码学原语参数、系统参数和动态参数的集合，供接下来的协议和算法使用。

密钥生成

（PKeyGen(epp) → (pk, sk)）：

  • 输入： 系统参数 epp。

  • 输出： 密钥对 (pk,sk)，其中 pk是公共密钥，sk是私钥。

    步骤解释：

    • 平台通过调用盲签名方案 Πbs中的密钥生成算法 KeyGen(epp)，生成密钥对
    • 平台还会初始化其内部状态 st，其中包括但不限于：
      • 注册用户集 USet=∅：表示系统开始时没有用户。
      • 标识符集 ID=∅：表示系统开始时没有任何用户标识符。
    • 为了简化，公共密钥 pk将被隐式作为用户和平台后续协议和算法的输入。

用户注册

注册请求与用户注册（⟨Join(reqjoi), Issue(sk, st)⟩ → (uid, Rdreg /⊥; b, st′)）

  • 输入：
    • reqjoi：包含所有注册所需的信息，特别是用户的真实身份和银行信息等用于合规性的资料。这些信息将用于身份验证过程。
    • sk：平台的私钥，用于对某些敏感操作进行签名或验证。
    • st：平台的内部状态，包含已注册用户的信息、交易记录、余额等。
  • 输出：
    • uid：用户在系统中注册成功后生成的唯一标识符（Unique ID）。每个用户在系统中都有一个唯一的ID。
    • Rdreg /⊥：这是注册记录（或失败标志）。
      • Rdreg：如果注册成功，平台将为用户生成一个注册记录 Rdreg，其中包括用户的访问令牌（access token），该令牌用于之后的身份验证。令牌的属性 Rdreg.cred被设置为 tk，是用户身份验证的凭证。
      • ⊥：如果注册失败，返回符号 ⊥，表示注册未成功。
    • b：表示注册是否成功的标志。
      • b = 1：注册成功。表示用户成功注册，并且系统生成了一个新的用户记录 Rdreg。平台的内部状态会更新，记录包括用户的元数据、后续交易、余额等信息。
      • b = 0：注册失败。
    • st′：更新后的平台内部状态，包括注册新用户的信息。例如，用户的标识符 uid、交易记录、余额等都将更新。

平台存款

存款操作过程（⟨Deposit(uid, Rdreg, reqdep), Credit(sk, st)⟩ → (; b, st′)）

  • 输入：
    • uid：用户的唯一标识符，表示该用户在平台中的身份。
    • Rdreg：用户的注册记录或访问令牌，包含了用户的认证凭证，用于身份验证。
    • reqdep：存款请求，包含存款的具体信息，如存款的金额、资产类型等。
    • sk：平台的私钥，用于执行与存款操作相关的加密或签名操作（如果需要）。
    • st：平台的当前内部状态，包含所有注册用户的账户信息、余额、交易记录等。
  • 输出：
    • b：表示存款交易是否成功的标志。
      • b = 1：表示存款交易成功。
      • b = 0：表示存款交易失败。
    • st′：更新后的平台内部状态，包含了存款后的用户余额、存款交易记录以及相关的元数据等。

用户认证（Authentication）

  • 用户首先通过提供其 uid 和 Rdreg（即注册记录或访问令牌）来进行身份验证。这是用户登录过程的一部分，目的是确保只有授权用户才能进行存款操作。
  • 在认证成功后，用户可以继续进行存款操作。

提交存款请求（Deposit Request）

  • 用户提交存款请求（reqdep），该请求包含所有与存款相关的必要信息。例如：
    • 存款金额：用户打算存入的资产数量。
    • 资产类型：存款的资产类型（如某种加密货币或法定货币）。
    • 可能还有其他必要的信息，如存款方式、来源账户等。

平台验证并处理存款

  • 平台使用 sk（私钥）和 st（平台的当前状态）来处理存款请求。平台会执行以下操作：

    1. 验证请求：平台首先验证存款请求的合法性，例如确认存款金额是否有效、资产类型是否受支持、以及是否满足其他必要的条件。
    2. 更新用户账户：如果请求有效，平台会将存款金额添加到用户的账户余额中，更新用户的资产状况。
    3. 记录存款交易：平台会将该存款操作作为交易记录保存，以便后续查看和审计。

    在这些操作成功完成后，平台返回标志 b = 1，表示存款交易成功。

更新平台状态

  • 更新后的平台状态（st′）：存款交易完成后，平台会更新其内部状态 st′

    。这个状态包含以下内容：

    • 用户余额更新：用户存款后，其余额会增加，平台将存款金额添加到用户的账户中。
    • 交易记录：存款操作将被记录下来，成为用户的交易历史的一部分。
    • 相关元数据：可能包括存款的时间、交易ID等元数据，用于追踪和管理交易。

成功与失败的处理

  • 如果存款请求成功且平台更新了状态，b = 1 表示交易成功，用户可以查看更新后的余额和交易记录。
  • 如果存款请求无效或出现错误（如金额无效、资产类型不受支持等），则 b = 0 表示交易失败，平台可能不会对状态进行任何更改。

资产交换

普通交换（plain exchange）、私人交换（private exchange）后者准备用于未来的匿名提现

资产交换操作（⟨Exchange(uid, Rdreg, reqexc), Update(sk, st)⟩ → (Rdast; b, st′)）

  • 输入：
    • uid：用户的唯一标识符，用于在系统中标识该用户。
    • Rdreg：用户的注册记录或访问令牌，包含认证凭证，用户登录所需的凭证。
    • reqexc：用户提交的资产交换请求，包含了交换的类型（普通交换或私人交换）、涉及的资产、交换金额等。
    • sk：平台的私钥，用于对敏感信息进行签名或验证。
    • st：平台的当前内部状态，包括用户的账户余额、交易记录、资产状态等。
  • 输出：
    • Rdast：用户在交易后的资产记录，记录了交换的具体信息（例如，资产标识符、金额、签名等）。
    • b：表示交易是否成功的标志。
      • b = 1：交易成功，平台更新了用户的资产记录和余额。
      • b = 0：交易失败，平台没有进行任何更新。
    • st′：更新后的平台内部状态，包括用户余额的更新、交换交易的记录以及相关的元数据等。

交换操作类型（reqexc.op）

交换操作有两种类型，具体由 reqexc.op 参数决定：

  • pln（普通交换）：用户和平台之间进行普通的资产交换，平台知道交换的金额、资产类型等信息。
  • prv（私人交换）：用户进行私人交换，平台不能知道交易的具体金额，交换的金额是保密的。私人交换通常是为了未来的匿名提现准备。

账户余额状态：

  • 用户登录系统时，提供其 uid（用户唯一标识符）和 Rdreg（注册记录或认证令牌），系统会根据用户的身份信息返回该用户的账户状态。
  • 如果该用户 之前提取过某种资产（例如资产 i），那么用户的资产余额状态（balance state）会分为两部分：
    • bali（普通余额，plain balance）：这是用户账户中未被承诺的普通余额，表示用户的资产 i 在平台中的明文余额。用户可以直接使用这部分余额进行交易或其他操作。
    • {com_j_i}（承诺值集，committed values）：这是一组承诺值，用于表示用户已提取的资产 i 的部分余额。这些承诺值是通过加密承诺机制生成的，平台无法直接看到这些资产的具体金额。每个 com_j_i 对应用户在提取资产 i 时所做的一个承诺。承诺的数量为 l，即用户关于资产 i 的所有提取操作的总数。

l 的含义

  • l：这是一个整数，表示用户与资产 i相关的承诺的总数。也就是说，用户对资产 i 所做的承诺次数是 l，每个承诺都被记录为一个 com_j_i
    • 每个 com_j_i 是一个承诺值，通常采用加密哈希或加密承诺函数（例如：Pedersen承诺）来生成，以保证资产数额的隐私性。
    • 这些承诺值保证了用户的交易和资产信息的隐私性，平台在没有知道资产具体数额的情况下，也可以验证用户是否有足够的余额进行交易。
  • bali（普通余额）：是平台能看到的余额，通常是透明的，用户可以直接使用这部分资产进行普通交换或其他操作。
  • {com_j_i}（承诺值集）：是隐私的，平台看不到具体金额，但通过这些承诺，平台可以验证用户是否有足够的资产用于交换，而无需知道具体的余额。例如，在私人交换操作中，用户提交的是承诺值而不是实际金额，这保证了用户的隐私。

已经提取过了资产i，明文余额+l个承诺的已经提取的资产余额

普通交换（plain exchange）

  • 用户提交普通交换请求，reqexc = (i, ki, j, kj, pln)其中：

    • i：用户要交换的资产类型。
    • ki：用户想要交换的资产数量。
    • j：用户交换的目标资产类型。
    • kj：目标资产的数量。

  • 用户必须证明其账户中有足够的余额进行交易：

    • 用户需要证明资产 i 的余额 bali 大于或等于交换金额 ki。
    • 用户通过生成零知识证明（ZKAoK）来证明这一点：
      • 零知识证明包含了用户已承诺的资产数额 {com_j_i} 和金额 ki 是否小于或等于账户余额 bali。
      • 证明 bali - ki 大于等于所有承诺的总和（使用明文交换）

  • 用户将交换请求和证明发送给平台。如果证明有效，平台会：

    1. 更新用户的账户余额 bali ← bali - ki。
    2. 更新目标资产余额 balj ← balj + kj。
    3. 记录交易，并更新平台状态 st′。

    这里就是使用ki的金额交换对方kj的金额

私人交换（private exchange）

  • 用户提交私密交换请求 reqexc = (i, ki, prv)，表示希望将 ki 单位的资产 i 转换为私密版本。
  • 用户对交换金额 ki 生成一个新的承诺 com*，并生成一个零知识证明 π，证明其余额足够进行交换。
  • 如果盲签名验证成功，平台更新用户的资产记录，添加新的承诺 com*，记录私密交换交易及相关元数据。用户获得新的资产记录 Rdast。

匿名取款过程

匿名操作:

  • 用户不需要登录其账户，而是以匿名访客的身份进行操作。平台无法识别与其交互的用户身份。

请求和资产记录:

  • 用户准备取款请求 reqwit = (i, ki, meta) 和资产记录 Rdast。

  • reqwit 包含资产类型 i、取款数量 ki 和元数据 meta。meta 可能包括链上地址和其他交易所需的元数据。

  • Rdast 是用户的资产记录，格式为 (rid, i, ki, σ)。

发送请求:

  • 用户将 (reqwit, Rdast) 直接发送给平台。

平台验证:

  • 平台解析资产记录 Rdast，检查 rid 是否在 ID 集合中。

  • 如果 rid 不在 ID 中（表示该资产记录从未被使用过），并且盲签名验证 Πbs.Vrfy(rid, i, ki, σ) = 1 成功，平台进行链上转账到 meta 中指定的地址。

交易成功:

  • 如果验证成功，平台输出 b = 1，表示交易成功，并更新内部状态，包括将 rid 添加到 ID 集合中，以防止重复使用。

交易失败:

  • 如果 rid 已经在 ID 中或盲签名验证失败，用户和平台中止交易。

关键点

  • 匿名性: 用户不需要登录账户，平台无法识别用户身份，确保了交易的匿名性。

  • 防止重复使用: 通过检查 rid 是否在 ID 集合中，平台可以防止资产记录的重复使用。

  • 盲签名验证: 使用盲签名验证确保交易的合法性和安全性。

客户合规操作过程

操作目的:

该操作用于确保用户遵循税务合规要求。用户需要为特定时间段内的交易进行税务申报。

用户登录:

用户使用其用户ID uid 和访问令牌 Rdreg 登录系统。

获取状态信息:

用户从系统的内部状态 st 中获取合规状态、余额状态和交易历史。

计算承诺:

用户基于交易历史和余额状态，计算一个承诺 c。承诺的计算涉及用户的真实身份 id、总成本 cp1、总收益 cp2 和时间段 mt。

承诺的格式为 c = Πbs.Com(id, cp1, cp2, mt; r)，其中 r 是用于承诺的随机数。

生成证明:

用户生成一个证明，证明其身份、总成本、总收益和时间段计算的正确性。

平台签名:

平台对用户提交的文件进行签名，表示对用户提交的合规文件的认可。

结果输出:

如果操作成功，平台输出文档 doc 和成功标志 b = 1，并更新内部状态 st′。

如果操作失败，平台输出失败标志 b = 0。

关键点

合规性: 该操作确保用户遵循税务合规要求，通过提交正确的交易记录和计算结果。

隐私保护: 使用承诺和证明技术，用户可以在不泄露具体交易细节的情况下，证明其合规性。

平台认可: 平台通过签名来认可用户提交的合规文件，确保文件的合法性和完整性。
