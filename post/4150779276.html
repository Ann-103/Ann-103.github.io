匿名凭据

匿名凭证：允许用户在不泄露其他信息的情况下证明自己拥有凭证。即使验证者和发行人勾结，他们也无法了解凭证发放给用户的身份，也无法了解如何以及在哪里使用凭证。

对于匿名凭证的理想安全属性：

  • 不可伪造 unforgeability
  • 正确 correctness
  • 不可链接 unlinkability

由于匿名本身造成的问题：

  • 女巫攻击 sybil ：用户生成大量凭证并进行非法买卖
  • 追踪审计和撤销

note：这里是DID凭证系统本身就存在的问题？

构建匿名凭证的传统方法是使用盲签名，即发证机构签名时并不知道具体的凭证内容。eg：Pointcheval-Sanders 签名方案。

签名：

  • 密钥生成

选择参数：选择一个大素数 p 和一个生成元 g

私钥：选择一个随机数 x作为私钥，满足 1<x<p−1

公钥：计算公钥$ y=g^x\pmod p$

  • 消息哈希：对待签名的消息 m进行哈希，得到哈希值 H(m)。

  • 随机数选择：选择一个随机数 k，满足 1<k<p−1,且 k与 p−1互质。

  • 计算签名：

    计算$ r=g^k\pmod p$

    计算$ s=k^{−1}(H(m)+x⋅r) mod(p−1)$

    签名输出：签名为 (r,s)

验证：

  • 检查 r和 s是否在有效范围内，即 0<r<p和 0<s<p−1。
  • 计算$ w=s^{−1}mod(p−1)$
  • 计算$u_1=H(m)⋅w \pmod{p−1} 和 u_2=r⋅w \pmod{p−1} $
  • 计算 v=(gu1⋅yu2)(modp)v=(g^{u_1}⋅y^{u_2})\pmod pv=(gu1​⋅yu2​)(modp)，与r对比

基于区块链可以避免单一发行方单点故障问题，并且不需要签名密钥。发证阶段区块链作为透明账本存放公开可验证的匿名凭证，验证阶段用户需要提供证明自己拥有某个匿名凭证。

DAC：在去中心化匿名凭证方案中，凭证保存在某种形式的透明日志中，这可以是集中式的、分布式的或通过拜占庭系统或区块链以去中心化的方式操作。虽然这种方法消除了凭证颁发中的一个障碍，但具体协议在性能和操作上存在局限。例如，该协议要求所有客户端拥有已颁发凭证的完整列表，并未解决实际使用案例中的其他复杂性。

Coconut：为了减少对颁发者的信任，方案探索了阈值颁发和多个颁发者的支持。虽然这在有多个愿意的颁发者时改善了情况，但并未解决愿意或能够部署新技术的颁发者稀缺的问题。此外，它也未能提供协调不同身份文件格式或多个颁发者使用案例的手段。

Decentralized Anonymous Credentials NDSS 2014

C. Garman, M. Green, and I. Miers. Decentralized anonymous credentials. In NDSS. Citeseer, 2014.

传统匿名凭证系统，例如盲签名需要一个可信的凭证发行者，这个中心化的发行者存在单点故障风险，容易成为攻击目标。在点对点网络等临时网络中难以部署,因为没有可以完全信任的单一机构。希望颁发凭证的过程（如果不是身份本身的建立）不再依赖于受信任方。

提到了如果无法确定链上地址是否与相同的真实身份相关联，攻击者可以通过生成大量地址来获得不成比例的利益

创新方案:

  • 基于区块链，提出了一个新的匿名凭证方案，不需要可信的凭证发行者

凭证系统可以按需实例化，并由一组不信任的对等体操作，进一步展示了对系统的扩展，以创建可更新的（例如，有状态的）匿名凭证，其中用户根据其身份属性的变化获得新的凭证。

  • 去中心化直接匿名认证。我们展示了如何去中心化直接匿名证明协议，允许自组织或分布式系统中的单个节点集合安全地断言其系统状态的属性。我们的方法消除了由一方管理DAA凭据的需要，从而消除了现有系统的主要限制。我们提供了一个去中心化（dDAA）结构的示例性描述。

  • 可审计凭证。我们的技术也可用于扩展现有的集中式证书系统，允许对已颁发的证书进行公开审计。这有助于防止凭证颁发者受到攻击，并允许网络轻松检测和撤销不适当的凭证授予。

传统的匿名凭证系统有两种类型的参与者：用户和组织。每个用户都有一个密钥skU，彼此和组织都知道他们的假名（例如用户A在组织O中的化名是NymAONym^O_ANymAO​），去中心化匿名凭证没有一方代表组织，而是被强制执行特定凭据颁发策略并协同维护迄今为止颁发的凭据列表的用户法定人数所取代。为了与之前的工作保持一致，我们保留了“组织”一词。分布式匿名凭证系统由全局事务分类账、一组事务语义以及以下（可能是概率性的）算法组成：

Form：选择随机数sknym=rsk_{nym}=rsknym​=r，生成假名Nym=g0rgsk1Nym=g^r_0g^{sk_1}Nym=g0r​gsk1​

一旦被网络接收，所有各方都可以使用MintVerify例程和验证辅助数据所需的任何外部程序来验证凭证和身份断言的正确性。该过程可以直接由网络节点执行，也可以在事后由单个凭证验证器进行验证。

当用户希望向某个验证者显示凭证时，她首先扫描公告板，以获得一组CO，其中包含属于特定组织的所有候选凭证。

接下来，她使用MintVerify例程验证每个凭证（如果她还没有这样做），并验证辅助身份认证信息。然后，她运行Show算法生成凭证，并将其直接传输给验证器。验证器还收集CO中的凭据集，并使用ShowVerify例程验证凭据。如果此例程输出1，则她接受证书认证。

Coconut NDSS2019

Coconut是一种新型的选择性披露证书方案，支持分布式阈值发布、公共和私有属性、重新随机化和多个不可链接的选择性属性披露。

Coconut与区块链集成，以确保机密性、真实性和可用性，即使凭证颁发机构的一个子集是恶意的或离线的。

我们为Chainspace和以太坊实现并评估了一个通用的Coconut智能合约库；并介绍了三种与匿名支付、电子请愿和分发抗审查代理相关的应用程序。Coconut使用短而计算高效的凭据，我们的评估表明，大多数Coconut加密原语平均只需要几毫秒，验证所需时间最长（10毫秒）。

zk-cred

  • 通用零知识证明协议，证明在MPT树上有证书（承诺）且属性符合访问标准。灵活访问标准定制，支持隐藏属性等匿名凭证功能。凭证可公开审计，区块链维护凭证列表，撤销只需要从Merkle树中移除凭证

1）消除了凭证颁发者持有签名密钥的需要：凭证可以颁发给实例化为透明度日志、拜占庭系统甚至区块链的公告板；

2） 将现有身份证件转换为匿名证件，而无需修改证件或与发证机关协调；

3） 允许在多个凭据上使用灵活、可组合和复杂的身份声明。具体来说，在使用护照匿名访问年龄限制视频的真实场景中，使用zk-cred进行身份断言所需的时间不到150毫秒。

贡献

  • 隐私保护的身份验证工具包：zk-creds 是一个用于隐私保护身份验证协议和匿名凭证的工具包，提供灵活的身份声明，并且不需要可信的发行者。与以往的工作相比，zk-creds 的一个关键贡献是使用通用的零知识证明，而不是基于盲签名的定制证明系统。
  • 范式转变：将通用零知识证明作为匿名凭证的基础，而不是盲签名，代表了一种范式转变。zk-creds 允许在协议设计和部署后，仍然能够实现完全的隐私和表达能力。使用 zk-creds 构建的单一方案可以适应不断变化的需求，而无需开发新的定制加密协议。此外，应用特定的逻辑可以通过多种公开可用的工具使用简单的编程语言定义和修改，方案的实例化由编译器处理。
  • 灵活的访问标准：通用零知识证明使 zk-creds 支持灵活和可组合的访问标准。用户不仅可以私下展示其凭证满足某些任意访问标准检查，还可以在任何时间（即使在系统设置或凭证发行后）由任何方定义这些标准，并动态组合成小工具。这种灵活性使 zk-creds 能够满足现实世界身份验证机制的需求，因为要求和使用案例可以随时动态变化
  • 支持现有身份基础设施：zk-creds 的第二个主要贡献是支持现有的政府身份基础设施，而无需修改或合作。通过使用通用零知识证明，我们可以将越来越多包含在国家身份证和护照中的数字（非匿名）身份信息转换为匿名凭证。这种支持文档使我们能够提供数字化的方式，类似于走进美国机动车辆管理局（DMV）并出示现有身份文件以获取驾照，而无需向发行者暴露任何信息
  • 公开可验证的凭证：zk-creds 还支持公开可验证的凭证。由于发行的凭证列表可以由公共系统（如透明日志或区块链）维护，并且每个凭证可以包括证明其发行的支持文档，因此所有发行凭证的集合是公开可审计的。这在凭证可以通过签名密钥秘密发行时是不可行的。因此，我们只需信任发行者将凭证（及其支持文档）添加到列表中，任何妥协或不当行为都是可检测和可逆的。

总结贡献点：

  • 通过可重用证明显著提高现有去中心化方案的性能，ShowCred 的执行时间小于 150 毫秒；
  • 支持现有的物理身份文件（如护照），无需修改；
  • 提供灵活和可组合的小工具，能够在系统设置后组合以表达复杂的访问标准检查；
  • 允许已发行凭证的公开审计，而不损害匿名性；
  • 提供了一种新机制（盲 Groth16），用于私下链接多个零知识证明，从而实现证明的重新随机化和重用；
  • 包括一个完整的应用程序，用于访问年龄限制的视频，具有克隆抵抗性，使用现有护照进行发行。

非目标：zk-creds 不涉及凭证的链上验证。该工作构建了可以在没有中央方持有签名密钥的情况下发行的灵活凭证（尽管我们也支持基于签名的发行）。这与工业和学术界的其他工作（例如 [52]）不同，后者考虑通过智能合约验证现有（即中心发行的）匿名凭证。关于匿名凭证的链上验证的问题并不是如何去除中心化发行，而是如何在智能合约执行成本极高的情况下最小化验证成本。通常通过在零知识证明中批量验证来降低验证成本，这适用于任何匿名凭证方案，包括这里提出的方案。

overview

1）凭证生成和发布：用户证明自己满足凭证的属性

cred :=Com(nk,rk, attrs; r)

nk：假名密钥，一个用于生成持久假名的private random value

rk：rate key，用于生成速率限制token的private random value

attrs ∈ A：一组公共和隐藏属性

r：承诺随机性（commitment randomness）

当凭证出现在issuer I的凭证列表CL上时，称I颁发了证书，这里的凭证列表可以用多种方式实例化，累加器、MT等，这里采用的方式是默克尔森岭，以及一组对应的默克尔树列表。

CL := (F, Tˉ\bar{T}Tˉ)，其中身份验证路径θ提供成员身份证明。

每个发行人都有对应的发行标准，例如cred中的出生日期与签名的数字护照相匹配。然后通过issuer认证通道，请求者（运行IssueReq）向发行者发送带有zk支持文档sd的cred。例如，Groth16证明或数字签名，发行人运行IssueGrant，成功后将cred添加到他们的列表中，并返回一个身份验证路径θ，证明其发行。

2）用户证明自己拥有已发布的某个匿名属性

访问标准（possible access criteria ）Φ := {ϕ | ϕ : A → {0, 1}}

这里Φ是指所有从A元素映射到01的函数集合，

在匿名通道上，用户通过提供零知识证明来显示凭证（运行ShowCred），证明他们拥有有效的已颁发凭证（θ作为私人见证witness），其见证中的属性w满足验证器的访问标准。该证明还必须绑定到某个会话上下文aux。verifier运行VerifyShow，成功后授予用户访问权限

3）简单协议流程（U是用户，I是凭证颁发者，V是验证者）

Setup(1λ1^λ1λ) → pp

  • 生成系统参数。此步骤生成所需的公共系统参数（pp），这些参数会用于后续的凭证和访问控制操作。λ 表示安全参数，通常与系统的安全级别（例如比特强度）相关。

IssueSetup(pp) → ι

  • 建立公共属性字段和凭证发行准则。使用之前生成的系统参数 pp，此步骤会设立一个公共属性集合（attrs），以及凭证发行的标准或准则（ι），这些准则将控制凭证如何被颁发。

ShowSetup(pp) → ϕ

  • 设定访问准则。此步骤设定一个新的“访问准则”（ϕ）。ϕ 是访问凭证的条件，定义了哪些情况下可以展示凭证。Φ 是所有可能访问准则的集合。

IssueReq_U(pp, ι, attrs, wsd, iauxsd) → (cred, sd)

  • 凭证申请。用户可以基于设定的发行准则（ι）和属性（attrs）请求一个凭证（cred），并生成相应的支持文档（sd）。这里 wsd 和 iauxsd 表示可能涉及的其他附加信息或支持文档。

IssueGrant_I(pp, ι, CL, cred, sd) → (CL′, θ)

  • 颁发凭证。此步骤决定是否根据申请（cred）及其支持文档（sd）授予用户请求的凭证。如果授予成功，凭证将被添加到凭证列表（CL）中，并返回凭证的证明（θ）。

ShowCred_U(pp, ϕ, CL, cred, θ, w, aux) → (πlink, aux)

  • 展示凭证。当用户需要展示其凭证时，通过检查其凭证是否满足设定的访问准则（ϕ）。此步骤会生成一个“展示链接”（πlink），证明用户持有符合准则的凭证，并附带额外的上下文信息（aux）。

VerifyShow_V(pp, ϕ, CL, πlink, aux) → b

  • 验证凭证展示。验证用户所展示的凭证是否符合设定的访问准则（ϕ），并检查相关上下文信息（aux）。返回一个布尔值 b，表示验证是否通过。

RevokeCred_I(pp, CL, cred) → CL′

  • 撤销凭证。如果某个凭证需要被撤销，此步骤会从凭证列表（CL）中删除该凭证，并返回更新后的凭证列表（CL′）。

Related

Decentralized Anonymous Credentials NDSS 2014：首次提出去中心化匿名证书的概念，消除单一发行人的假设和发行人持有密钥的需要。但是，他们要求用户在本地存储完整的凭证列表，并且计算证明时间和每次显示的已颁发凭证数量呈线性关系（即使是静态凭证列表）。

相比之下，zk-creds开发了一种新的方法和新的密码学（盲Groth16），以允许证明重用和快速凭证显示。而且，通过使用Merkle树，zk-creds要求用户只存储对数大小的见证数据来计算凭证显示。

他们使用的定制西格玛协议集不提供凭证和身份属性的组合，也不支持复杂的身份声明。

最后也是最重要的是，这项工作没有回答如何在不向发卡机构披露敏感信息的情况下做出颁发证书的决定的问题（即zk支持文档地址的问题）。

Coconut[55]匿名凭证方案通过阈值签名解决了发行挑战。在Coconut中，凭据由n个静态方在假设t>1/2是诚实的情况下颁发，但是必须有足够多的愿意发行凭证的参与方。如果在某些情况下，虽然有很多潜在的参与方愿意发行凭证，但由于某种原因，没有任何一个参与方被认为是可信的，那么阈值发行机制就无法解决发行者稀缺的问题。

Coconut只支持凭证中属性的选择性披露，而不支持对属性进行复杂的零知识证明。它不符合我们的灵活性或动态生成访问标准的设计目标。
